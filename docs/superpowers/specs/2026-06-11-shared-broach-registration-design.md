# 共通ブローチ所持登録と各計算機能への反映 設計

日付: 2026-06-11
ステータス: 承認済み

## 概要

ヘッダーメニューを再編して「各種登録」ドロップダウンを新設し、その配下に既存の「ラビットノート」と新規の「共通ブローチ」登録ページを置く。共通ブローチ（内部識別子: `SHARED_BROACHS`、全 26 種）の所持数を localStorage に登録し、スコア計算ページと編成組合計算ページで「所持ブローチ縛り」として利用できるようにする。

## 決定事項（ヒアリング結果）

| 論点 | 決定 |
|------|------|
| 新ページの役割 | 所持登録ページ（所持数を localStorage に登録） |
| 利用先 | スコア計算ページ + 編成組合計算ページの両方 |
| 登録粒度 | 種別ごとの所持数（0〜10） |
| フレンド枠の扱い | 所持制約の対象外。全 26 種から自由（同種 2 個も可）。「推奨ブローチ」として扱う |
| 未登録ユーザーの扱い | 「所持ブローチ縛り」トグルで切替。OFF なら従来どおり |
| 編成組合計算のトグル OFF 時 | 従来どおり共通ブローチなしで探索（既存の理論値結果と互換） |
| 表示名 | 「共通ブローチ」に統一（既存の「共有ブローチ」表記も修正）。内部識別子は `sharedBroach` のまま |
| 「所持衣装」メニュー | トップレベルのまま（検索機能も兼ねるため移動しない） |

## ゲーム仕様の前提

- 固有ブローチ（`FixedBroach`、カード紐付き）は付け替え不可。本件の所持登録の対象外
- 共通ブローチは UR カードのみ装備可。固有ブローチ持ち UR は 1 個、それ以外の UR は 2 個まで
- デッキは 6 枠（slotIndex 0=センター, 1-4=メンバー, 5=フレンド）。フレンド枠は他人のカードのため、所持ブローチが影響するのは自チーム 5 枠 × 最大 2 個 = **最大 10 個**
- 共通ブローチの効果は装着カードに依存しない（`teamBuilder.ts` で確認済み）。条件付きブローチ（S/B/M 属性枚数分 +300）もデッキ全体の属性枚数で決まる。カードに依存するのは装備可否（枠数）のみ

## フェーズ分割

各フェーズを独立した PR・リリースとする。

| フェーズ | 内容 | 規模 |
|---------|------|------|
| Phase 1 | メニュー再編 + 共通ブローチ登録ページ + 用語統一 + localStorage キー追加 | 小〜中 |
| Phase 2 | スコア計算ページに「所持ブローチ縛り」トグル | 中 |
| Phase 3 | 編成組合計算に所持ブローチ割当を組み込み | 大 |

## Phase 1: 登録基盤

### データモデルと localStorage

- `STORAGE_KEYS` に `SHARED_BROACH_COUNTS: 'i7_shared_broach_counts'` を追加
- 形式: `Record<string, number>`（キー = `SharedBroach.id` の文字列、値 = 所持数）。`i7_card_counts` と同じモデル
- 所持数の入力上限は **10**（使用上限が 10 個のため、それ以上は計算に影響しない）。0 は「未所持」でキーごと削除
- `FooterTools` のバックアップは `STORAGE_KEYS` 起点のため、キー追加だけでエクスポート/インポート対象に含まれる

### ヘッダーメニュー再編（`HeaderNav.svelte`）

- `items` 配列のトップレベル「ラビットノート」を削除し、同位置に `DropdownItem`「各種登録」を追加。子は「ラビットノート」(`/rabbit-note/`) と「共通ブローチ」(`/shared-broach/`)
- ドロップダウン開閉 state を汎用化: `scoreCalcOpen` / `scoreCalcWrapper` 等の単一ドロップダウン前提のハードコードを廃し、`openDropdown: string | null`（ラベルをキー）+ ラッパー要素の Map で複数ドロップダウンに対応する。モバイル側も同様にラベルキーで開閉管理
- 「所持衣装」はトップレベルのまま維持

### 共通ブローチ登録ページ

- 新規ページ `src/pages/shared-broach/index.astro` + `src/components/SharedBroachEditor.svelte`
- `RabbitNoteEditor` と同じ構成（ビルド時プリレンダリング + クライアント JS、変更は即時 localStorage 保存）
- `SHARED_BROACHS` 全 26 種を ALL / Shout / Beat / Melody / 条件付き のグループ別に一覧表示。各行にステータス値と所持数入力（−/+ ボタン付き数値入力、0〜10）

### 用語統一

- ユーザー可視テキストの「共有ブローチ」を「共通ブローチ」に統一（`DeckSlots.svelte` の選択肢ラベル、`max-score-finder/index.astro` の説明文等）
- 内部識別子（`sharedBroach` / `SHARED_BROACHS` / URL `shared-broach`）は命名規約どおり変更しない
- `CLAUDE.md` の用語ポリシー・命名規約に「表示名は『共通ブローチ』、内部識別子は `sharedBroach`」を追記

## Phase 2: スコア計算ページへの反映

- 「所持ブローチ縛り」チェックボックスを追加（既存トグル類と同じ UI パターン）。状態は `i7_score_calc_state` に永続化
- ON のとき:
  - 自チーム枠（slotIndex 0〜4）の共通ブローチ選択肢を「所持数 −（他スロットでの使用数）> 0」のものに制限。現在選択中の値は常に選択肢に残す
  - フレンド枠（slotIndex 5）は従来どおり全 26 種・同種重複可
  - 保存デッキ・共有 URL から所持数超過の編成を読み込んだ場合は自動クリアせず、警告メッセージの表示のみ（データを壊さない）
- OFF のとき: 完全に従来どおり

## Phase 3: 編成組合計算への反映

- 「所持共通ブローチを割り当てる」トグルを追加（既存の「所持衣装縛り」とは独立）。OFF = 従来どおりブローチなし探索
- ON のとき、候補編成ごとに解析的グリーディ割当を行う:
  1. 自チーム 5 枠の装備可能数 k を算出（UR かつ固有ブローチなし = 2、固有ブローチあり UR = 1、UR 以外 = 0。最大 10）
  2. 各所持ブローチの寄与値を楽曲の属性重みで評価（条件付きブローチは編成の属性枚数を掛ける）し、上位 k 個を所持数の範囲で貪欲に割当
  3. フレンド枠は全 26 種から最良 2 個を「推奨ブローチ」として割当（同種重複可・所持制約なし）
- ブローチの組合せ総当たりはしない（効果が装着カード非依存のため不要。探索空間は増えず、1 編成あたりの評価コストがわずかに増えるのみ）
- 割当ロジックは新規純粋モジュール `src/lib/score/broachAssignment.ts` に切り出す（UI / Worker 両方から import。`maxScoreFinder.ts` と同じパターン）
- 結果表示（`SearchResults` / 衣装詳細）に割当ブローチを表示。スコアは割当込みで算出
- 前提検証: スコアがチーム属性値に対して線形であることを実装時に確認する。線形ならグリーディが厳密最適。非線形要素があれば設計を再検討し ADR に追記する

## エッジケース

- 所持ブローチ合計 < k → あるだけ割り当てる
- 登録ゼロでトグル ON:
  - スコア計算: 自チーム枠は選択肢なし + 「共通ブローチが未登録です」の案内（登録ページへのリンク付き）を表示
  - 編成組合計算: 同様の案内を表示し、探索はブローチなしで実行
- 条件付きブローチの寄与値は編成依存のため、編成ごとに毎回評価する

## テスト

- Vitest 単体テスト: `broachAssignment.ts`（容量計算・グリーディ割当・条件付きブローチ・所持数制約）、スコア計算の選択肢制限ロジック
- Playwright E2E: 登録ページの入力・永続化、ヘッダーナビのドロップダウン動作（既存テストへの追加）
- 各フェーズで `npm run dev` + スクリーンショット確認 → リリースノート更新 → PR
