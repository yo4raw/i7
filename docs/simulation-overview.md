# スコアシミュレーションの仕組み

> 本ドキュメントは「なぜシミュレーションをしているのか」と「実装上で何が起きているのか」を、
> 実装変数名に依存せず自然言語で説明することを目的とする。
> 厳密な仕様は `docs/score_calc_spec.md`、コードは `src/lib/score/engine.ts` を参照。

## 1. なぜシミュレーションするのか

多くのスキルは **確率で発動** するため、同じデッキ・同じ楽曲でもライブごとに最終スコアが変動する。
このため、デッキ評価には以下の両方が必要になる:

- **理論値 (最低・最高)**: 全スキル不発 / 全スキル発動 というシナリオで取り得る両端
- **分布 (ばらつき)**: 確率を踏まえた「実際にはどれくらいのスコアが出やすいか」

理論値だけではスキル発動確率の違い (40% と 90%) を比較できず、平均値だけでは「最悪ケースでどの程度落ちるか」が分からない。
シミュレーションは既定 100 回（UI から任意回数指定可、`MC_ITERATIONS = 100`）ライブを繰り返したときのスコア分布を算出することで、
**平均値・中央値・標準偏差・最悪/最良ケース** をまとめて提示する。

## 2. 全体像

```
┌───────────────┐   ┌───────────────┐   ┌──────────────────────┐
│ デッキ 6 枠    │ → │ チーム属性値  │ → │ ノート列を先頭から    │
│ (各カード + ブローチ) │   │ (Shout/Beat/Melody) │   │ 末尾まで 1 回走査     │
└───────────────┘   └───────────────┘   │ (= 1 回分のライブ)    │
                                         └──────────┬──────────┘
                                                    ↓ 既定 100 回繰り返し
                                         ┌──────────────────────┐
                                         │ N 個のスコア列        │
                                         └──────────┬──────────┘
                                                    ↓ 統計処理
                                         ┌──────────────────────┐
                                         │ 最小/最大/平均/中央値/ │
                                         │ 標準偏差/90%点/ヒスト │
                                         └──────────────────────┘
```

## 3. 入力

- **デッキ**: センター・メンバー1〜4・フレンドの 6 枠
- **楽曲**: ノート配列 (ノートごとに「属性・始点/終点・ライトグループ」を持つ)
- **固定ブローチ**: 属性加算 or スコア直接加算
- **オプション**: スコアアップアシスト (+20%)、スコアアップバッジ (+15% など)、スキルレベル

## 4. 1 回のライブ (内部的な 1 試行) で何が起きるか

楽曲のノートを先頭から末尾まで、1 ノートずつ処理する。
各ノートの処理タイミングで、全カードのスキル発動判定が同時並行的に進む。

### 4-1. タイマースキル (例: JokerFlag2 四葉環)

| 挙動 | 内容 |
|------|------|
| 発動判定単位 | 秒 (楽曲の時間経過ベース) |
| 発動チャンス回数 | `floor(楽曲の長さ / スキルのカウント秒)`。例: 104 秒 ÷ 16 秒 = **6 回** |
| 判定タイミング | 16 秒目・32 秒目・…・96 秒目 |
| 判定方式 | 各タイミングで確率 `per%` ロール (例: 47%) |
| 発動時の効果 | そのタイミングに最も近いノートに **value 点を 1 回加算** (例: +7200) |

Lv5 スキルが全発動すれば `6 × 7200 = 43,200 点` が理論最大加算。
期待値は `6 × 7200 × 47% = 20,304 点`。

### 4-2. コンボスキル (例: 屋外フェス2 逢坂壮五)

| 挙動 | 内容 |
|------|------|
| 発動判定単位 | ノート数 (コンボ経過ベース) |
| 発動チャンス回数 | `floor(総ノート数 / スキルのカウント)`。例: 428 ÷ 16 = **26 回** |
| 判定タイミング | 16 ノート目・32 ノート目・…・416 ノート目 |
| 判定方式 | 各タイミングで確率 `per%` ロール (例: 46%) |
| 発動時の効果 | 判定ノートに value 点を **1 回加算** (例: +6403) |

Lv5 スキルが全発動すれば `26 × 6403 = 166,478 点` が理論最大加算。
期待値は `floor(26 × 6403 × 46%) = 76,579 点`。

### 4-3. 判定縮小 (Perfect) スキル (例: 記念日2024 四葉環)

| 挙動 | 内容 |
|------|------|
| 発動判定単位 | ノート数 (ただし楽曲先頭の除外区間は判定対象外) |
| 先頭除外数 | `excludeHead = max(notes_20 グループのノート数, デッキ内縮小スキル count の最小値)`。例 (MONSTER GENERATiON + count=20 スキル): `max(21, 20) = 21` |
| 発動チャンス回数 | `floor((総ノート数 − excludeHead) / スキルのカウント)`。例: `floor(407 / 20)` = **20 回** |
| 判定方式 | カウント分のノートを通過するたびに確率 `per%` ロール (例: 40%) |
| 発動時の効果 | `value` 秒の持続時間中、判定対象ノートのスコアが `rate` 倍になる。`rate` はカードデータの `skill.rate` を参照（スキルレベル依存、UI 既定は Lv5） |
| 複数枚同時 | キューイング仕様により発動区間は時間軸上で重ならない。各スキルの「発動回数 × 持続秒」を **単純加算** し、`songDuration` で 100% キャップ。独立事象 OR `1 − Π(1−p_i)` は **使わない** (`docs/shrink-skill-spec.md §1-1, §4`) |

**スキルレベル別 `rate` の典型値**:

| スキル Lv | rate | 追加倍率 (`rate − 1.0`) |
|----------|-----:|----------------------:|
| Lv1 | 1.2 | +0.2 |
| Lv2 | 1.3 | +0.3 |
| Lv3 | 1.4 | +0.4 |
| Lv4 | 1.5 | +0.5 |
| Lv5 | 1.6 | +0.6 |

持続時間中は各ノートに `floor(アシスト後素点 × (rate − 1.0))` が追加される (Lv5 / rate=1.6 で +0.6 倍)。
先頭除外区間のノートは縮小効果の対象外。

### 4-4. 1 ノートあたりのスコア式

各ノートのスコアは以下を合算:

```
per_note = floor(属性値 × ライト倍率 × ノートレート)   ← 基本スコア
         + floor(per_note × 0.6)                       ← 縮小持続中のみ
         + Σ (このノートに乗ったタイマー発動 value)
         + Σ (このノートで発動したコンボスキル value)
```

### 4-5. 試行終了後

1 回分の総スコア = 全ノートのスコア合計 + ブローチによる固定加算 + (バッジ + アシスト)% 加算。

## 5. 試行を既定 100 回繰り返す

- 既定で `MC_ITERATIONS = 100` 回実行（UI から任意回数指定可）
- 乱数は `XorShift128Plus` 方式。シードを固定すれば結果は常に同一 (決定論的)
- UI をブロックしないよう `MC_CHUNK_SIZE = 50` 回ごとに非同期ブレイクを挟む

## 6. 得られる統計値

N 個のスコアから以下を算出（既定 N=100）:

| 名称 | 意味 |
|------|------|
| **実測最小 / 実測最大** | N 回中の最低 / 最高スコア |
| **平均** | N 回の単純平均 (期待スコア) |
| **中央値** | スコアを並べて真ん中に来る値 |
| **標準偏差** | スコアのばらつきの大きさ |
| **90%点 (p90)** | N 回中 90% のスコアがこの値以下 |
| **ヒストグラム** | スコア分布の可視化用 |
| **カード別統計** | 各カードのスキル平均発動回数・平均スコア寄与 |

## 7. 理論値との関係

シミュレーションとは別に、以下も同じ入力から計算される:

| 値 | 定義 | 用途 |
|------|------|------|
| **理論最小** | 全スキルが一度も発動しなかった場合 | 下限の保証 |
| **理論最大** | 全スキルが毎回発動した場合 | 上限の保証 |
| **算術期待値** | `発動回数 × 確率 × 効果値` を足し合わせた単純期待値 | 外部サイト (例: オタライフ) との比較 |

実測スコアは必ず `[理論最小, 理論最大]` の区間内に収まる。
算術期待値は MC 平均と近い値になるが、分散情報は持たない。

## 8. シミュレーションが「分散」を見るのに向いている理由

算術期待値では「発動確率 40% を 20 回判定するスキル」と「発動確率 90% を 10 回判定するスキル」が
同じ期待加算点でも、**実際に出るスコアのばらつきは前者のほうが大きい**。
シミュレーションは試行を重ねることで両者の `stddev` / `p90` / `mcMin` に差を出す。

例: 判定縮小スキル 1 枚と 2 枚で平均は上がるが、単純加算カバー率が 100% キャップに達すれば
stddev は下がる (最悪ケースも底上げされる)。この挙動は算術期待値では見えない。
