# 0004: 共通ブローチ所持登録とスコア計算・編成探索への反映方針

- 日付: 2026-06-11
- ステータス: 承認（2026-06-11 ユーザー承認）

## 背景

共有ブローチ（`SHARED_BROACHS`、全 26 種）はこれまで「所持」の概念がなく、スコア計算では全種から自由選択、編成組合計算では一切未使用（未装着固定）だった。ユーザーが実際に所持しているブローチの範囲で編成を検討できるようにしたい。

## 決定

1. **ヘッダーメニュー再編**: トップレベルの「ラビットノート」を「各種登録」ドロップダウンに置き換え、配下に「ラビットノート」「共通ブローチ」を置く。「所持衣装」は検索機能を兼ねるためトップレベルのまま
2. **所持登録**: 新ページ `/shared-broach/` で種別ごとの所持数（0〜10）を `i7_shared_broach_counts` に登録する。上限 10 は使用上限（自チーム 5 枠 × 2 個）に合わせた値
3. **表示名の統一**: ユーザー可視テキストは「共通ブローチ」とする（ゲーム内表記に準拠）。内部識別子は `sharedBroach` / `SHARED_BROACHS` のまま
4. **スコア計算**: 「所持ブローチ縛り」トグル（OFF がデフォルト）。ON 時は自チーム 5 枠（slotIndex 0〜4）の選択肢を残り所持数で制限する。フレンド枠（slotIndex 5）は他人のカードのため対象外（全種・同種重複可）。所持数超過の保存デッキ等は自動クリアせず警告表示のみ
5. **編成組合計算**: 「所持共通ブローチを割り当てる」トグル（OFF がデフォルト）。OFF は従来どおりブローチなし探索（既存の理論値結果と互換）。ON 時は候補編成ごとに**解析的グリーディ割当**を行う:
   - 共有ブローチの効果は装着カードに依存しない（条件付きブローチもデッキ全体の属性枚数で決まる。`teamBuilder.ts` で確認済み）ため、ブローチ組合せを総当たりする必要がない
   - 自チームの装備可能枠数 k（UR かつ固有ブローチなし = 2 / 固有ブローチあり UR = 1 / UR 以外 = 0）を求め、楽曲ノーツから算出した属性重みでブローチ単価を評価し、上位 k 個を所持数の範囲で貪欲割当
   - フレンド枠は全 26 種から最良ブローチを容量分割当（推奨ブローチ、所持制約なし）
   - スコア計算の `calcNoteScore` はノーツごとの floor を含むため厳密には非線形だが、属性値に対して単調非減少であり、グリーディ割当は「floor を無視した線形近似での厳密最適」。誤差はブローチ寄与値に対して無視できる水準

## 検討した代替案

- **ブローチ組合せの総当たり**: 装備パターンが多重集合で爆発し（26 種 × 最大 10 個）、探索時間が非現実的。効果が装着カード非依存である以上、総当たりは不要なため却下
- **編成組合計算の OFF 時を「全ブローチ自由割当」にする**: 理論値が従来結果から変わってしまうため却下（ユーザー指定で従来互換を維持）
- **所持/未所持フラグのみの登録**: 同一種を複数装備できるかを表現できないため却下（所持数登録を採用）
- **「所持衣装」も「各種登録」配下へ移動**: 所持衣装ページは検索機能を兼ねるためトップレベル維持（ユーザー指定）

## 影響

- 新 localStorage キー `i7_shared_broach_counts`（`STORAGE_KEYS` 経由でバックアップ対象に自動包含）
- `HeaderNav.svelte` のドロップダウン開閉 state を単一前提からラベルキー管理に汎用化
- Phase 分割: (1) 登録基盤 → (2) スコア計算反映 → (3) 編成組合計算反映 の 3 PR で段階リリース
